# SmartAgent4 AIRI 前端舞台集成 — 产品规格说明

> **迭代轮次**：第八轮（AIRI 前端角色舞台）
> **最后更新**：2026-04-13

## 产品愿景

在 SmartAgent4 现有车机中控界面（Cockpit）中，新增一个可视化的角色舞台模块，使系统已有的情感标签、动作指令和语音合成输出能够被一个可动的 2D Live2D 角色实时承接和表演。最终目标是让用户在与 SmartAgent4 对话时，不仅能看到文字回复和 emoji 标注，还能看到一个具有表情变化、肢体动作和说话口型联动的虚拟角色，形成"对话即表演"的沉浸式交互体验。

## 目标用户

本次迭代的主要用户是 SmartAgent4 的车机端使用者（主驾/副驾）。他们在驾驶或乘坐过程中通过语音或文字与系统交互，期望获得更直观、更有温度的反馈。次要用户是项目开发者和演示人员，他们需要一个可调试、可观测的角色舞台来验证情感输出链路的正确性。

## 核心功能（MVP）

### 功能 F1：角色舞台容器组件（AiriStageContainer）

- **描述**：在 Cockpit 页面中间区域（当前为车辆背景图位置）嵌入一个 React 组件，作为 Live2D 角色的渲染容器。容器需要自适应布局，不影响现有的顶栏、右侧助手面板和底部卡片。
- **用户故事**：作为车机用户，我希望在中控屏中间看到一个虚拟角色，以便感受到更生动的交互体验。
- **验收标准**：
  1. 当 Cockpit 页面加载完成时，中间区域应显示一个默认的 Live2D 角色模型，角色处于闲置（idle）状态。
  2. 当浏览器窗口大小变化时，角色容器应自适应缩放，不出现溢出或裁剪。
  3. 当 Live2D 模型资源加载失败时，容器应显示优雅的降级提示（如角色头像占位符），不阻塞页面其他功能。
- **优先级**：P0（高）

### 功能 F2：舞台事件协议与前端消费层（StageEventBus）

- **描述**：定义一套前端可消费的角色状态事件协议，包含 `expression`（表情）、`motion`（动作）、`tts_lipsync`（口型联动）、`idle_state`（闲置状态）四类事件。后端在 `chat.sendMessage` 返回时，除了文本回复外，还需附带结构化的舞台事件数据，前端通过事件总线分发给角色渲染器。
- **用户故事**：作为车机用户，我希望虚拟角色的表情和动作能与 AI 回复内容同步变化，以便感受到角色"真的在回应我"。
- **验收标准**：
  1. 当后端返回包含 `[expression:smile]` 标签的回复时，前端应解析出 `expression` 事件并分发到事件总线，角色应切换为微笑表情。
  2. 当后端返回包含 `[animation:nod]` 标签的回复时，角色应播放点头动作。
  3. 当后端返回包含 `[pause:1.5]` 标签的回复时，角色应在对应时间点暂停动作。
  4. 当后端返回不包含任何情感标签的纯文本时，角色应保持中立（neutral）表情和闲置状态。
- **优先级**：P0（高）

### 功能 F3：表情驱动与切换（ExpressionDriver）

- **描述**：将 `expression` 事件映射为 Live2D 模型的表情参数变化。支持 SmartAgent4 已定义的 7 种基础情感（neutral、happy、sad、angry、surprised、fearful、disgusted），以及前端 emotionParser 中扩展的 9 种情感（think、shy、love、proud、worried、confused、excited、relieved、默认降级）。每种情感对应一组 Live2D 参数（眼睛、眉毛、嘴巴等），支持强度（intensity）控制。
- **用户故事**：作为车机用户，我希望角色在 AI 表达不同情感时能展现对应的面部表情，以便更直观地理解 AI 的情绪状态。
- **验收标准**：
  1. 当收到 `expression: happy, intensity: 0.8` 事件时，角色应展示明显的微笑表情。
  2. 当收到 `expression: sad, intensity: 0.3` 事件时，角色应展示轻微的悲伤表情（强度较低）。
  3. 当连续收到不同表情事件时，角色应平滑过渡（过渡时间 200-500ms），不出现突变。
  4. 当收到未知表情类型时，角色应降级为 neutral 表情，并在控制台输出警告。
- **优先级**：P0（高）

### 功能 F4：动作播放与状态管理（MotionDriver）

- **描述**：将 `motion` 事件映射为 Live2D 模型的预设动作播放。支持的动作包括：点头（nod）、挥手（wave）、摇头（shake_head）、鞠躬（bow）、歪头（head_tilt）等。动作播放完毕后自动回到闲置状态。支持动作优先级，高优先级动作可打断低优先级动作。
- **用户故事**：作为车机用户，我希望角色在 AI 表达动作指令时能做出对应的肢体动作，以便增强交互的生动感。
- **验收标准**：
  1. 当收到 `motion: nod` 事件时，角色应播放点头动作，播放完毕后回到闲置状态。
  2. 当收到 `motion: wave` 事件时，角色应播放挥手动作。
  3. 当动作播放过程中收到更高优先级的动作事件时，当前动作应被打断，立即播放新动作。
  4. 当收到未知动作名称时，应忽略该事件并在控制台输出警告，不影响角色当前状态。
- **优先级**：P0（高）

### 功能 F5：说话态与基础口型联动（LipsyncDriver）

- **描述**：当 TTS 音频播放时，角色应进入说话状态（speaking），嘴巴参数随音频节奏变化，形成基础的口型联动效果。TTS 播放结束后，角色回到闲置状态。本阶段采用基于音量电平的简化口型驱动，不要求精确的音素级口型同步。
- **用户故事**：作为车机用户，我希望角色在"说话"时嘴巴能动起来，以便感受到角色确实在与我对话。
- **验收标准**：
  1. 当 TTS 音频开始播放时，角色嘴巴参数应开始变化，形成张合效果。
  2. 当 TTS 音频播放结束时，角色嘴巴应回到闭合状态。
  3. 当用户未点击"生成语音"（即无 TTS 数据）时，角色不应进入说话状态。
  4. 口型变化的频率和幅度应与音频节奏大致匹配，不出现明显的不同步。
- **优先级**：P1（中高）

### 功能 F6：闲置状态与呼吸动画（IdleStateManager）

- **描述**：当角色未接收到任何表情、动作或说话事件时，应自动进入闲置状态，播放轻微的呼吸动画和随机的微小动作（如眨眼、轻微摇晃），使角色看起来"活着"。支持多种闲置子状态：idle（默认闲置）、thinking（AI 正在处理中）、listening（用户正在语音输入）。
- **用户故事**：作为车机用户，我希望角色在等待时也有轻微的动态效果，以便感受到角色是"活的"而不是静态图片。
- **验收标准**：
  1. 当页面加载完成且无对话进行时，角色应自动播放呼吸动画和随机眨眼。
  2. 当用户发送消息后、AI 回复到达前，角色应切换为 thinking 状态（如歪头、手托下巴）。
  3. 当用户开启麦克风进行语音输入时，角色应切换为 listening 状态（如微微前倾、注视前方）。
  4. 当新的表情或动作事件到达时，闲置动画应被平滑打断。
- **优先级**：P1（中高）

### 功能 F7：AIRI Bridge 状态面板（BridgeStatusPanel）

- **描述**：在 Cockpit 页面中提供一个可折叠的 AIRI Bridge 连接状态面板，显示当前连接状态、消息计数、活跃角色 ID 等信息。开发者可通过此面板手动连接/断开 AIRI Server，以及切换角色配置。该面板复用后端已有的 `airi.status`、`airi.connect`、`airi.disconnect` tRPC 路由。
- **用户故事**：作为开发者，我希望能在界面上看到 AIRI Bridge 的连接状态，以便快速排查角色舞台不工作的原因。
- **验收标准**：
  1. 当 AIRI Bridge 处于 `ready` 状态时，面板应显示绿色指示灯和"已连接"文字。
  2. 当 AIRI Bridge 处于 `disconnected` 或 `failed` 状态时，面板应显示红色指示灯和错误信息。
  3. 当用户点击"连接"按钮时，应调用 `airi.connect` 路由并更新状态显示。
  4. 当用户点击"断开"按钮时，应调用 `airi.disconnect` 路由并更新状态显示。
- **优先级**：P1（中）

## 非功能范围

本次迭代明确不包含以下内容：

- **VRM/3D 角色渲染**：3D 表现层属于第二阶段增强，本次仅实现 2D Live2D 方案。
- **精确音素级口型同步**：本阶段采用基于音量的简化口型驱动，不实现 viseme 级别的精确同步。
- **多角色同屏**：本阶段仅支持单角色显示，不支持多角色场景。
- **角色编辑器**：不提供用户自定义角色外观或动作的编辑功能。
- **AIRI Server Runtime 的部署与管理**：前端舞台独立于 AIRI Server Runtime 运行，不依赖外部 AIRI 服务即可展示角色。
- **车载传感器数据集成**：情境感知的情感分发属于车载情感化优化计划的范畴，不在本次迭代中。

## 技术栈

| 组件 | 技术选型 | 选型理由 |
|------|----------|----------|
| Live2D 渲染引擎 | pixi-live2d-display + PixiJS v7 | AIRI 开源仓库已验证的成熟方案，社区活跃，与 React 集成方便 |
| 前端框架 | React 19 + TypeScript | 与现有 SmartAgent4 前端技术栈一致 |
| 状态管理 | Zustand | 轻量级状态管理，适合角色舞台的事件驱动模式 |
| 事件总线 | 自定义 EventEmitter（mitt） | 极小体积（200B），用于舞台事件分发 |
| 音频分析 | Web Audio API（AnalyserNode） | 浏览器原生 API，用于提取音量电平驱动口型 |
| 构建工具 | Vite | 与现有项目一致 |
| 后端通信 | tRPC | 与现有项目一致，复用 `airi.*` 路由 |

## 关键假设

1. Live2D Cubism SDK for Web 可以在项目中正常使用（需确认许可证兼容性）。
2. 默认角色模型资源（`.moc3` + `.model3.json` + 纹理）可以从 AIRI 开源仓库或 Live2D 官方示例中获取。
3. 现有 `emotionParser.ts` 中的标签解析逻辑可以直接复用，作为舞台事件的数据源。
4. 后端 `chat.sendMessage` 的返回值可以扩展，在不破坏现有前端兼容性的前提下附加舞台事件数据。
5. 浏览器端 Live2D 渲染的性能开销在车机硬件上可接受（目标帧率 30fps 以上）。

## 已知风险

| 风险 | 影响程度 | 发生可能性 | 缓解策略 |
|------|----------|------------|----------|
| Live2D Cubism SDK 许可证限制 | 高 | 中 | 优先使用免费版 SDK；如受限，考虑使用开源 Live2D 替代方案（如 CubismWebFramework） |
| 默认角色模型资源缺失 | 中 | 低 | 从 Live2D 官方示例（Haru、Hiyori）获取免费模型；AIRI 仓库中也有可用资源 |
| 车机端浏览器性能不足 | 高 | 中 | 提供性能降级模式（降低渲染帧率、简化模型）；设置 `enableAiriStage` 开关 |
| 表情/动作映射不完整 | 中 | 中 | 建立映射缺失的降级机制（未知表情→neutral，未知动作→忽略）；持续补充映射表 |
| 前后端事件协议变更 | 中 | 低 | 采用版本化的事件协议；前端对未知事件类型做容错处理 |

## 成功标准

1. Cockpit 页面中间区域成功显示一个可动的 Live2D 角色，角色具有呼吸和眨眼等闲置动画。
2. 当 AI 回复包含情感标签时，角色能在 500ms 内切换到对应表情，且过渡平滑。
3. 当 AI 回复包含动作标签时，角色能正确播放对应动作，播放完毕后回到闲置状态。
4. 当 TTS 音频播放时，角色嘴巴能随音频节奏变化，形成基本的说话效果。
5. 角色舞台的引入不影响现有 Cockpit 页面的功能（对话、会话管理、记忆卡片等）。
6. 新增模块的单元测试覆盖率不低于 80%。
