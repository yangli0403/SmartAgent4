## 一、能力总览与代码依据

### 1.1 「文件大师 / 文件整理」是否还在？

**具备。** 当前代码在 `SmartAgentApp.initialize()` 中调用 `registerFileOrganizerTools()`，将下列 **4 个内置工具**注册到全局 `ToolRegistry`（与高德、网易云等 MCP 并列，不依赖远程 MCP URL）：

| 工具名 | 用途 |
|--------|------|
| `analyze_directory` | 分析目录文件分布 |
| `find_duplicates` | 查找重复/同名文件 |
| `delete_files` | 安全删除文件 |
| `move_files` | 批量移动文件 |

`server/agent/agent-cards/fileAgent.json` 中已将上述工具与能力标签 `file_organization`、`duplicate_detection` 声明给 **fileAgent**。  
**注意**：传统「文件搜索 / 打开 / 创建」等依赖 **本地 stdio 文件系统 MCP** 的工具，需在 `mcp-config.json` 中启用并配置 `local-filesystem` 后才可用；**文件整理大师四工具为内置注册，与该项独立。**

### 1.2 记忆系统（技能化）

见 **第三章**（与 0402 一致）：主动存储、检索、更新/遗忘、自动提取开关、边界与综合场景。

### 1.3 其他领域（与 v1.5 对齐、项目名改为 SmartAgent4）

## 三、记忆系统技能化测试（0402 全文并入）

> 以下与 **0402 版** 第一～七章一致，便于单文件打印与验收。

### 3.1 系统架构与改造概述

SmartAgent4 在本次迭代中完成了记忆系统从"被动捕获"到"主动调度"的范式转变。系统通过引入**记忆技能化（Skills-based Memory）**理念，将记忆的定义权和执行权交还给 Agent，彻底解决了多轮任务上下文割裂、提取时机僵化以及模糊指代处理困难等问题。

**核心改造点**：

1. **主动记忆工具**：`memory_store`、`memory_search`、`memory_update`、`memory_forget` 四个内置工具。
2. **System Prompt 策略注入**：任务总结（Store）、模糊消解（Search）、状态更新（Update/Forget）。
3. **降级自动提取**：默认关闭每轮自动 LLM 提取（受 `AUTO_EXTRACTION_ENABLED` / `MEMORY_AUTO_EXTRACTION` 等开关控制，以代码与环境变量为准）。
4. **后台服务兼容**：`source: "agent_skill"` 与巩固 / 主动引擎管道兼容。

### 3.2 主动存储与任务总结测试用例

**测试目标**：验证 Agent 能否在多轮任务后主动调用 `memory_store` 总结并存储。

| 编号 | 测试场景 | 测试步骤与输入 | 预期行为与结果 | 涉及工具 |
|------|---------|--------------|--------------|---------|
| STORE-001 | 单轮事实记忆存储 | 用户："我叫李华，住在北京朝阳区。" | Agent 主动调用 `memory_store` 存入事实记忆（`type: "fact"`），回复确认已记住。 | `memory_store` |
| STORE-002 | 单轮偏好记忆存储 | 用户："我以后只喝无糖的咖啡。" | Agent 主动调用 `memory_store` 存入偏好记忆（`type: "preference"`），回复确认偏好。 | `memory_store` |
| STORE-003 | 多轮导航任务总结 | 1. "我要回家" → 2. 补充家地址或规划 → 3. "途径一下山姆超市" → 4. 规划路线 | 最后一步应主动 `memory_store` 将完整情景写入（如回家路线途径山姆）。 | `maps_direction_*` → `memory_store` |
| STORE-004 | 多轮行程预订总结 | 查高铁 → 订票 → 订酒店等 | 任务完成后 `memory_store` 总结行程。 | `memory_store` |

### 3.3 主动检索与模糊消解测试用例

| 编号 | 测试场景 | 测试步骤与输入 | 预期行为与结果 | 涉及工具 |
|------|---------|--------------|--------------|---------|
| SEARCH-001 | 模糊指代（路线） | 前置已存路线记忆；用户："把昨晚的导航路线再发我一遍。" | 主动 `memory_search` 并回复。 | `memory_search` |
| SEARCH-002 | 模糊指代（地点） | 前置餐厅记忆；用户："上次那家餐厅叫什么？" | `memory_search` 后准确回答。 | `memory_search` |
| SEARCH-003 | 隐式偏好 | 前置饮食偏好；用户："推荐几家附近的餐厅。" | 先 `memory_search` 再结合地图推荐。 | `memory_search` → `maps_around_search` |
| SEARCH-004 | 无结果降级 | 询问无记录的事实 | 坦诚无记录并引导补充。 | `memory_search` |

### 3.4 状态更新与记忆修正测试用例

| 编号 | 测试场景 | 测试步骤与输入 | 预期行为与结果 | 涉及工具 |
|------|---------|--------------|--------------|---------|
| UPDATE-001 | 事实变更 | 已存住址；用户："我搬到上海浦东了。" | `memory_search` → `memory_update`。 | `memory_search` → `memory_update` |
| UPDATE-002 | 偏好变更 | 已存旧偏好；用户声明新偏好 | 检索后 `memory_update`。 | `memory_search` → `memory_update` |
| FORGET-001 | 明确要求遗忘 | 用户要求删除某类记忆 | `memory_search` → `memory_forget`。 | `memory_search` → `memory_forget` |
| FORGET-002 | 纠错 | 用户指出记忆错误 | `memory_update` 或删后重建。 | `memory_search` → `memory_update` |

### 3.5 降级开关与工作记忆测试用例

| 编号 | 测试场景 | 测试步骤与输入 | 预期行为与结果 | 验证方式 |
|------|---------|--------------|--------------|---------|
| SWITCH-001 | 默认关闭自动提取 | `MEMORY_AUTO_EXTRACTION` 未设或 `false`，多轮闲聊 | 不触发旧式全量提取流水线（以日志为准）。 | 日志无批量 "Extracted …" |
| SWITCH-002 | 开启自动提取 | `MEMORY_AUTO_EXTRACTION=true` | 恢复异步提取与行为检测（以日志为准）。 | 日志有提取相关输出 |
| WORKMEM-001 | 工作记忆连续性 | 先报坐标，再要求「附近有什么」 | 后续工具使用上一轮坐标。 | 工具参数含坐标 |

### 3.6 异常与边界处理测试用例

| 编号 | 测试场景 | 预期行为与结果 |
|------|---------|----------------|
| EDGE-001 | `memory_store` 缺 `content` | 明确错误提示，Agent 不崩溃。 |
| EDGE-002 | 非法 `type` | 枚举校验错误提示。 |
| EDGE-003 | `importance`/`confidence` 越界 | 裁剪到合法区间或报错（以实现为准）。 |
| EDGE-004 | `memory_update` 无更新字段 | 错误提示。 |
| EDGE-005 | DB 异常时 `memory_search` | 友好错误信息，进程不崩溃。 |

### 3.7 端到端综合测试场景（0402）

**场景 A：偏好建立 → 闲聊 → 模糊调用「老规矩」**  
验证 `memory_store` + `memory_search` 与行程工具联动。

**场景 B：车牌录入 → 换车更新 → 限行查询**  
验证 `memory_store` → `memory_search` → `memory_update` 链路与下游工具。

**0402 环境要求（摘录）**：分支含记忆技能化提交；PostgreSQL；推荐较强遵循能力的 LLM。

## 八、文件系统与「文件整理大师」测试用例

### 8.1 文件整理大师（内置四工具，推荐优先测）

**前置**：准备**非系统关键路径**的测试目录（如用户文档下 `SmartAgent4_FileTest`），避免误删。

| 编号 | 场景 | 测试步骤（示例话术） | 预期 |
|------|------|---------------------|------|
| ORG-001 | 目录分析 | 「请分析一下文件夹 `D:\...\SmartAgent4_FileTest` 里文件分布」 | 调用 `analyze_directory`，返回统计/分布信息 |
| ORG-002 | 重复文件 | 在测试目录放入两个内容相同的文件，「查一下这个目录里有没有重复文件」 | 调用 `find_duplicates`，列出重复项 |
| ORG-003 | 批量移动 | 「把测试目录里所有 `.tmp` 移到子文件夹 `archive`」 | 调用 `move_files`，路径变更符合预期 |
| ORG-004 | 安全删除 | 「只删除测试目录里明确叫 `delete_me.txt` 的文件」 | 调用 `delete_files`，仅目标被删 |

**验证要点**：后端日志出现对应工具名；fileAgent 可被分类命中；删除类用例务必使用隔离目录。

### 8.2 传统文件操作（依赖 MCP）

若需测试 **search_files / open_file / list_directory** 等：在 `mcp-config.json` 启用 **local-filesystem** 并保证 `local-mcp-server` 路径可用，再复用 v1.5 中「文件系统」相关预期。**未启用时**仅测 **8.1 文件整理大师**即可代表「文件域」能力的一部分。
