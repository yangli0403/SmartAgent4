# AIRI 集成与优化计划（0412）

**作者：Manus AI**  
**日期：2026-04-12**

## 一、文档目的

本文对 0412 当天围绕 **AIRI 接入、Web 舞台展示、情感与动作承接、VRM/3D 扩展价值** 的分析内容进行统一整理，目标是把此前分散在多个文档与问答中的判断，收束为一份可直接归档、可继续指导实现的优化计划文档。

从当前项目状态看，SmartAgent4 已经完成了与 AIRI Runtime 的后端桥接基础，但尚未形成浏览器端的角色舞台呈现层。因此，这份文档的重点不是重复说明“有没有接口”，而是进一步回答三个工程上更关键的问题：第一，AIRI 开源仓库中哪些代码值得复用；第二，怎样以较低代价把角色舞台补进 SmartAgent4；第三，若当前阶段的基本底线是**必须稳定承接 SmartAgent4 的各类情感和动作输出**，那么最合理的技术路径是什么。[1] [2] [3]

## 二、当前状态的统一判断

本次整理后的统一结论可以概括为一句话：**SmartAgent4 已有 AIRI Bridge 后端接入能力，但前端角色舞台基本尚未开始建设。**

已有实现主要集中在 `server/airi-bridge/`，其职责包括与 AIRI Server Runtime 建立 WebSocket 连接、映射情绪与动作语义、处理音频包以及在聊天结果返回后向 AIRI 侧转发内容。这意味着，系统在“语义驱动层”和“运行时桥接层”上并不是空白，甚至已经具备比较清晰的情绪与动作输出协议基础。[1] [4]

但与此同时，当前前端目录下并没有发现成型的 `Live2D`、`VRM`、`Pixi`、`Three.js` 等浏览器端角色渲染实现，也没有成套的角色资源装载链路。因此，当前项目并不能直接在 SmartAgent4 现有 Web 页面里显示一个可动的 AIRI 角色。缺失的不是“后端能不能发消息”，而是“浏览器有没有角色舞台渲染器去消费这些状态”。[2] [4]

| 维度 | 当前状态 | 判断 |
| --- | --- | --- |
| AIRI Runtime 连接 | 已有 | 已可完成后端桥接与转发 |
| 情绪/动作语义映射 | 已有 | 已具备基础驱动语义层 |
| Web 端角色舞台 | 缺失 | 需要新增前端角色渲染模块 |
| 模型资源与加载链路 | 缺失 | 需要补齐默认角色资源与装载逻辑 |
| 前后端事件协议闭环 | 部分具备 | 还需补齐前端消费层 |

## 三、AIRI 开源仓库的复用价值

AIRI 开源仓库并不是单一演示工程，而是一个已经把舞台能力模块化拆开的 monorepo。对 SmartAgent4 最有价值的，不是照搬整个 Web 应用，而是有选择地复用其中已经成熟的舞台组件与运行时逻辑。

从现有判断看，`apps/stage-web` 更适合作为完整参考实现，用于观察 AIRI 在浏览器中的整体交互方式；`packages/stage-ui` 提供了共享业务层与状态组织方式；`packages/stage-ui-live2d` 则是当前最值得优先复用的部分，因为它更接近一个聚焦 Live2D 场景、资源加载与预览逻辑的独立能力包；`packages/stage-ui-three` 则代表未来向 VRM/3D 扩展时的主要能力来源。[2] [5]

| 模块 | 主要用途 | 对 SmartAgent4 的复用建议 |
| --- | --- | --- |
| `apps/stage-web` | 浏览器端完整舞台应用 | 参考实现，不建议整仓嵌入 |
| `packages/stage-ui` | 共享状态与交互层 | 按需裁剪借鉴 |
| `packages/stage-ui-live2d` | Live2D 渲染与加载逻辑 | **优先复用，适合当前首阶段** |
| `packages/stage-ui-three` | VRM/Three.js 3D 舞台能力 | 作为第二阶段增强 |

因此，本次整理后的建议依旧明确：**可以大量复用 AIRI 代码，但应采用“模块级复用”，而不是“应用级整体搬运”。** 这是控制成本和耦合边界的关键。

## 四、为什么不建议直接整体搬运 AIRI Web

从工程实现角度看，AIRI 当前 Web 端与 SmartAgent4 的前端技术栈并不一致。AIRI 的现有舞台 Web 更接近 Vue 体系，而 SmartAgent4 当前主体是 React + TypeScript + Vite 结构。若直接搬运完整 Web 应用，不仅要处理渲染框架差异，还会带入其路由、状态管理、样式体系与部分面向 AIRI 自身生态的业务约束。

这意味着，整体搬运虽然在理论上可行，但会让“快速验证角色展示效果”这个本来边界清晰的目标，演变成一项高耦合的双前端栈整合工程。相比之下，更合理的方式是在 SmartAgent4 中新增一个相对独立的 `airi-stage` 模块，把 AIRI 的 Live2D/VRM 运行逻辑作为能力源，由 React 侧做适配与包装。这样既能继承 AIRI 已有积累，又能把影响范围控制在单一功能域之内。[2]

## 五、当前阶段的基本底线：完整承接情感与动作输出

用户提出的要求非常关键，即当前完成效果的**基本底线**不是“先把 3D 做出来”，而是**必须稳定承接 SmartAgent4 中各类情感与动作输出**。这一定义实际上直接决定了技术优先级。

如果从“能否承接情感与动作输出”这个目标倒推，那么第一阶段并不要求先上 VRM/3D。原因在于，2D Live2D 方案已经足以完成大部分基础承接能力，包括表情切换、预设动作、说话状态、基础口型联动以及闲置/响应状态切换。真正决定底线是否成立的，不是渲染维度是 2D 还是 3D，而是系统内部是否定义出一套清晰、稳定、前后端一致的角色状态事件协议。[1] [4]

因此，当前阶段的首要任务应当是把 SmartAgent4 的多模态输出统一收束为前端角色舞台可消费的最小事件集合，而不是优先追求更重的表现层。换言之，**先保证“承接准确”，再追求“表现更强”。**

## 六、建议先固化的舞台事件协议

为了满足“情感和动作输出承接”的底线要求，建议先把当前系统中的角色相关输出整理为四类统一事件。这四类事件一旦稳定，前端展示层就可以独立迭代，而不需要反复修改上游逻辑。

| 事件类别 | 最小字段 | 作用 |
| --- | --- | --- |
| `expression` | `name`、`intensity`、`source` | 驱动表情切换与强弱变化 |
| `motion` | `name`、`priority`、`duration` | 触发预设动作与状态转换 |
| `tts_lipsync` | `speaking`、`audioLevel`、`segmentId` | 驱动说话态与基础口型联动 |
| `idle_state` | `state`、`reason`、`timestamp` | 维持闲置、等待、思考等过渡状态 |

在这一层之上，还可以保留几个辅助字段，例如 `characterId`、`connectionStatus`、`lastUtterance`、`emotionSource`。这些字段并不直接决定动作播放，但会极大简化角色切换、连接状态显示以及调试工作。[1] [4]

> 对当前 SmartAgent4 而言，真正需要优先完成的不是“哪个引擎更炫”，而是“任何一次上游情绪、动作或语音输出，都能在舞台层找到确定的落点并稳定执行”。

## 七、2D Live2D 路线为何适合作为第一阶段

如果把当前目标限定为“快速补齐角色舞台，并满足情感与动作承接底线”，那么 2D Live2D 路线是当前最合适的首阶段方案。原因不是它最先进，而是它与现阶段目标的匹配度最高。

首先，它能够较低成本地承接当前已有的 `expression`、`motion`、`speaking` 等状态语义。其次，AIRI 开源仓库已经为 Live2D 方向提供了可参考、可裁剪的现成模块，使得第一阶段无需从零构建浏览器舞台运行时。最后，在用户体验层面，2D Live2D 已经足够形成“角色确实在响应 SmartAgent4 输出”的产品感，这对于当前阶段的验证目标已经足够。[2] [5]

| 目标 | 2D Live2D 首阶段是否满足 | 说明 |
| --- | --- | --- |
| 表情承接 | 满足 | 可映射喜怒哀乐与强弱度 |
| 动作承接 | 满足大部分 | 可承接点头、挥手、待机等预设动作 |
| 语音联动 | 满足基础要求 | 可实现说话态与基础口型同步 |
| 产品展示效果 | 满足 | 已能形成可演示角色舞台 |
| 全身空间化表演 | 不完全满足 | 需 VRM/3D 扩展 |

因此，只要当前目标仍然是“在 SmartAgent4 里把情感和动作输出完整承接起来”，那么第一阶段上 2D 并不意味着妥协，而意味着采用最稳妥的落地路径。

## 八、VRM/3D 扩展真正额外带来的能力

VRM/3D 的价值并不在于“让系统从不能承接变成能承接”，而在于它让同一套情绪与动作协议拥有了更完整、更自然、更具空间感的表现力。

在 3D 路线上，角色不再只是切换脸部表情与少数二维动作，而是可以将同一条情绪指令扩展为更丰富的全身动作表现。例如高兴不仅可以体现在笑脸，还可以伴随抬手、前倾、身体重心变化；犹豫不仅是眉眼变化，还可以表现为低头、转身、视线偏移或肩部收缩。这种**全身骨骼动作表达能力**，才是 VRM/3D 相比 2D 的核心新增价值。[2]

| 表现层面 | 2D Live2D | VRM/3D 扩展后 |
| --- | --- | --- |
| 面部表情 | 强 | 更自然、更立体 |
| 预设动作 | 中 | 更丰富、可扩展为全身动作 |
| 语音联动 | 中 | 可加入头部、躯干、重心联动 |
| 空间表现 | 有限 | 强，适合沉浸感与数字人演示 |
| 性能与接入复杂度 | 较低 | 更高，需要更多资源与调优 |

所以，如果从路线设计上概括，结论应当是：**VRM/3D 是“表现力增强层”，而不是“承接能力底座”。** 当前真正的底座仍然是事件协议和舞台消费链路本身。

## 九、推荐的模块边界与目录思路

为了降低后续维护成本，建议在 SmartAgent4 中把 AIRI 舞台能力作为独立模块管理，而不是散落在现有业务组件中。整理后的推荐结构如下。

| 建议目录 | 作用 |
| --- | --- |
| `client/src/modules/airi-stage/` | React 侧 AIRI 舞台入口模块 |
| `client/src/modules/airi-stage/components/` | 舞台容器、状态面板、调试控件 |
| `client/src/modules/airi-stage/runtime/` | 舞台事件适配、资源管理、驱动封装 |
| `client/src/modules/airi-stage/assets/` | 默认角色资源与配置 |
| `vendor/airi-stage-live2d/` | 从 AIRI 裁剪出的 Live2D 能力模块 |
| `server/airi-bridge/` | 保留现有 Bridge 逻辑，继续负责上游转发 |

这一结构的重点在于把“复用来的 AIRI 代码”和“SmartAgent4 自己的业务代码”边界分开。这样后续无论是继续增强 Live2D，还是向 VRM/3D 迁移，都会更清晰、更容易演化。

## 十、建议的实施顺序

结合当前目标与项目现状，本次整理后的实施顺序建议分为三个层次。第一层先补协议，第二层再补舞台，第三层再做 3D 增强。

第一阶段，应优先把 `expression`、`motion`、`tts_lipsync`、`idle_state` 四类事件从现有 Bridge 与会话输出中稳定抽象出来，并确定默认角色、默认动作集与情绪映射关系。没有这一层，就算前端先接了模型，也很难形成稳定效果。

第二阶段，应在 SmartAgent4 前端新增 `AiriStage` 原型，用 2D Live2D 方式完成首个角色舞台版本。此时的关键验收标准应当是：**每一类上游情绪、动作和说话状态，都能在前端角色上被稳定承接并清晰展示。**

第三阶段，在前两层稳定后，再引入 VRM/3D 驱动器，把同一套事件协议映射为更丰富的全身动作与三维空间表现。这样可以确保第二阶段的沉淀不会被推翻，而是成为 3D 增强的稳定基础。

| 阶段 | 目标 | 产出 |
| --- | --- | --- |
| P0 | 固化角色状态事件协议 | 可稳定消费的情绪/动作/语音事件层 |
| P1 | 落地 2D Live2D 舞台 | 可演示、可联动的 Web 角色舞台 |
| P2 | 扩展 VRM/3D 表现层 | 更自然的全身动作与空间化演示 |

## 十一、最终结论

整理后的结论很明确。第一，AIRI 开源仓库中存在大量值得复用的前端舞台代码，尤其是 Live2D 与 VRM 方向的模块化能力，因此 SmartAgent4 的角色舞台并不需要从零开始写。第二，当前最合理的落地方式不是直接搬运 AIRI 的整个 Web 应用，而是在 SmartAgent4 中新增独立的 `airi-stage` 子模块，以模块级复用方式接入现成舞台能力。第三，现阶段的基本底线应当定义为：**先完整承接 SmartAgent4 的各类情感、动作与语音输出，再考虑更高维度的 3D 表现增强。**

基于这一结论，当前最推荐的路径仍是：**先用 2D Live2D 方案完成首阶段闭环，再把 VRM/3D 作为增强层纳入第二阶段。** 这一路线既满足当前的效果底线，也能最大程度控制接入成本与工程风险。
