# SmartAgent4 记忆系统优化 — 产品规格说明

## 产品愿景

SmartAgent4 已建立了完整的记忆系统架构骨架，但多项关键能力处于"已实现但未启用"或"已定义但未填充"的状态。本次优化的核心目标是**激活和连通已有的记忆能力**，通过生成 Embedding、启用混合检索、引入检索决策机制等手段，将记忆系统从"能用"提升到"好用"。同时，建立自动提取与 Agent 主动记忆的协同机制，恢复行为模式积累能力，并实现记忆置信度的动态演化，使系统具备自我修正的记忆管理能力。

## 目标用户

本次优化的直接用户是 SmartAgent4 的终端用户（与 AI 助手进行多轮对话的用户）。优化后，用户将体验到更精准的记忆召回（语义匹配替代文本匹配）、更低的响应延迟（无效检索被拦截）、更完整的个人信息记录（后台补漏提取）以及更可靠的记忆质量（置信度动态演化）。

## 核心功能（MVP）

### 功能 1：记忆写入时生成 Embedding（P0）

- **描述**：在 `addMemory` 函数中，对记忆内容调用 Embedding API 生成向量，并存入数据库的 `embedding` 字段。这是解锁向量检索能力的前提条件。
- **用户故事**：作为系统开发者，我希望每条新写入的记忆都自动生成 embedding 向量，以便后续的混合检索能够利用向量相似度进行语义匹配。
- **验收标准**：
  1. 当调用 `addMemory` 写入一条新记忆时，该记忆的 `embedding` 字段应被填充为有效的向量数组（非 null、非空数组）。
  2. 当 Embedding API 调用失败时，记忆写入操作不应被阻塞，`embedding` 字段可为 null，但应记录警告日志。
  3. 提供批量回填脚本，能够为历史记忆生成 embedding。
- **优先级**：P0

### 功能 2：contextEnrichNode 启用混合检索（P0）

- **描述**：修改 `contextEnrichNode` 中的 `getFormattedMemoryContext` 调用，传入 `useHybridSearch: true`，使前台记忆检索从纯文本匹配升级为 BM25 + 向量余弦相似度的双路召回。
- **用户故事**：作为终端用户，我希望系统能理解我的查询意图而非仅匹配关键词，以便在使用不同措辞时也能召回相关记忆。
- **验收标准**：
  1. 当用户查询与记忆文本不同但语义相似时（如查询"坚果"能召回"花生过敏"），检索结果应包含语义相关的记忆。
  2. 当 `embedding` 字段为空的记忆存在时，混合检索应优雅降级为纯 BM25 检索，不抛出异常。
  3. 检索性能不应因启用混合检索而产生超过 500ms 的额外延迟。
- **优先级**：P0

### 功能 3：引入 Pre-Retrieval Decision（P0）

- **描述**：在 `contextEnrichNode` 的记忆检索之前，增加一个快速的意图判断步骤，决定当前查询是否需要触发记忆检索。采用"规则 + 轻量 LLM"混合路径：先用规则快速判断明显的闲聊，再对不确定的查询调用 nano 级 LLM 做二分类。
- **用户故事**：作为终端用户，我希望系统在我闲聊时不会浪费时间检索记忆，而在我需要记忆支持时能快速响应。
- **验收标准**：
  1. 当用户输入明显的闲聊（如"谢谢"、"你好"、"哈哈"）时，系统应跳过记忆检索，直接进入后续流程。
  2. 当用户输入需要记忆支持的查询（如"我上次说我住在哪里"、"帮我推荐符合我口味的餐厅"）时，系统应触发记忆检索。
  3. Pre-Retrieval Decision 的判断延迟不应超过 200ms（规则命中）或 500ms（LLM 判断）。
  4. 决策结果应包含 `decision`（RETRIEVE / NO_RETRIEVE）字段，并在日志中记录。
- **优先级**：P0

### 功能 4：实现查询重写（Query Rewrite）（P1）

- **描述**：当 Pre-Retrieval Decision 判定需要检索时，利用当前对话历史将用户的模糊查询重写为自包含的明确查询。重写应在 Pre-Retrieval Decision 的同一次 LLM 调用中完成（当需要 LLM 判断时），以避免额外延迟。
- **用户故事**：作为终端用户，我希望系统能理解我话语中的代词和上下文引用，以便在我说"上次那家"时能正确找到对应的记忆。
- **验收标准**：
  1. 当用户输入包含代词的查询（如"那个"、"上次"、"他"）时，系统应结合对话历史将其重写为自包含的查询。
  2. 重写后的查询应同时用于 BM25 和向量检索路径。
  3. 当 Pre-Retrieval Decision 通过规则判断为 RETRIEVE 时（无 LLM 调用），查询重写应单独调用一次轻量 LLM。
  4. 重写后的查询应在日志中记录，便于调试和评估。
- **优先级**：P1

### 功能 5：建立自动提取与 Agent 主动的协同机制（P1）

- **描述**：保留 Agent 主动记忆作为主力，但增加两层保障：（1）在 Agent 调用 `memory_store` 时，自动执行置信度门控和去重校验（提取审计层）；（2）将自动提取改为"补漏模式"，在 DreamGatekeeper 触发做梦时对最近 N 轮对话进行批量回溯提取。
- **用户故事**：作为终端用户，我希望系统不会遗漏我提到的重要个人信息，即使 AI 助手在对话中忘记主动记录。
- **验收标准**：
  1. 当 Agent 调用 `memory_store` 写入重要性低于 0.3 的记忆时，审计层应拦截并返回警告信息。
  2. 当 Agent 调用 `memory_store` 写入与已有记忆高度相似（Jaccard > 0.6）的内容时，审计层应拦截或自动合并。
  3. 当做梦机制触发时，系统应对最近未被提取的对话进行批量回溯，提取出 Agent 遗漏的有效个人信息。
  4. 补漏提取的结果不应与已有记忆重复。
- **优先级**：P1

### 功能 6：解耦行为模式检测触发条件（P1）

- **描述**：将 `detectAndPersistPatterns` 的触发从"自动提取成功后"改为独立的事件驱动。在 `memoryExtractionNode` 中，无论自动提取是否开启，都基于工作记忆中累积的对话量（如每 10 轮对话）异步触发一次行为检测。
- **用户故事**：作为终端用户，我希望系统能持续学习我的行为习惯和偏好，即使在自动提取关闭的情况下也能积累行为模式数据。
- **验收标准**：
  1. 当 `MEMORY_AUTO_EXTRACTION` 为 `false` 时，行为模式检测仍能在达到对话轮数阈值后被触发。
  2. 行为模式检测的触发应基于工作记忆中的对话计数，而非依赖自动提取的执行结果。
  3. 检测到的行为模式应正确写入 `behavior_patterns` 表。
  4. 行为检测的触发不应阻塞主对话流程（异步执行）。
- **优先级**：P1

### 功能 7：实现 Confidence 动态演化（P1）

- **描述**：当用户再次提及与某条记忆一致的信息时，提升该记忆的置信度。当新提取的记忆与已有记忆属于同一 `versionGroup` 但内容不同时，降低旧记忆的置信度。在记忆写入（`addMemory`）和更新（`updateMemory`）流程中增加此逻辑。
- **用户故事**：作为终端用户，我希望系统能区分我确认过多次的信息和只提过一次的信息，以便在信息冲突时优先使用更可靠的记忆。
- **验收标准**：
  1. 当用户再次确认已有记忆的内容时（如已有"用户在上海工作"，用户再次说"我在上海上班"），该记忆的置信度应提升（增量 ≤ 0.15，上限 1.0）。
  2. 当用户提供与已有记忆矛盾的信息时（如已有"用户在上海工作"，用户说"我搬到北京了"），旧记忆的置信度应下降，新记忆应被写入。
  3. 置信度的调整应在日志中记录，包含调整原因和调整前后的值。
  4. 人格类型（personality）记忆的置信度不参与动态演化。
- **优先级**：P1

## 非功能范围

本次迭代明确不做以下事项：

- 不改变现有的数据库 schema 结构（`embedding` 字段已存在，无需新增列）
- 不引入新的外部数据库或向量数据库（继续使用 PostgreSQL 内置的向量能力）
- 不修改 Agent 的 System Prompt 策略（记忆技能的调用策略已在前序迭代中注入）
- 不实现 P2/P3 级别的优化（如 Reflect 推理集成、动态检索深度、语义聚类固化、做梦机制集成、预过滤增强、时效性自动设置）
- 不修改前端 UI 或客户端代码

## 技术栈

| 组件 | 技术选型 | 选型理由 |
|:---|:---|:---|
| 编程语言 | TypeScript 5.x | 现有项目语言，保持一致性 |
| 运行时 | Node.js 22.x | 现有项目运行时 |
| 测试框架 | Vitest | 现有项目测试框架，已有大量测试基线 |
| 数据库 | PostgreSQL + Drizzle ORM | 现有项目数据层，`embedding` 字段已定义 |
| Embedding API | OpenAI text-embedding-3-small | 通过已配置的 `OPENAI_API_KEY` 访问 |
| LLM（轻量级） | gpt-4.1-nano | 用于 Pre-Retrieval Decision 和查询重写 |
| 向量检索 | pgvector（PostgreSQL 扩展） | 利用现有 PostgreSQL 的向量能力 |

## 关键假设

1. PostgreSQL 数据库已安装 pgvector 扩展，支持向量相似度计算。
2. `OPENAI_API_KEY` 环境变量已配置，可用于调用 Embedding API 和轻量 LLM。
3. 现有的 `hybridSearch.ts` 中的向量检索逻辑是正确的，只需确保 `embedding` 字段被填充即可启用。
4. 现有的 `memories` 表 `embedding` 列的维度与所选 Embedding 模型的输出维度一致。
5. 做梦机制（`DreamGatekeeper` + `MemoryWorkerManager`）的触发逻辑是正确的，只需注入真实执行器。

## 已知风险

| 风险 | 影响程度 | 发生可能性 | 缓解策略 |
|:---|:---|:---|:---|
| Embedding API 调用延迟影响记忆写入速度 | 中 | 中 | 采用异步生成，不阻塞写入操作；失败时优雅降级 |
| 混合检索在大量记忆时性能下降 | 中 | 低 | 利用 pgvector 索引优化；设置合理的 limit 上限 |
| Pre-Retrieval Decision 的 LLM 调用增加延迟 | 中 | 中 | 规则层优先快速判断；LLM 使用 nano 级模型控制延迟 |
| 查询重写可能引入语义偏差 | 低 | 中 | 日志记录重写前后的查询，便于评估和调优 |
| 置信度动态演化的阈值设置不当 | 低 | 中 | 提供可配置的阈值参数；通过测试用例验证边界行为 |
| 补漏提取可能产生与 Agent 记忆的重复 | 中 | 中 | 在补漏提取前先查询已有记忆，执行去重校验 |

## 成功标准

1. 所有新写入的记忆均包含有效的 embedding 向量（覆盖率 > 99%，仅 API 故障时例外）。
2. 语义相似但文本不同的查询能成功召回相关记忆（混合检索命中率相比纯文本匹配提升 > 30%）。
3. 闲聊场景的记忆检索被有效拦截（Pre-Retrieval Decision 准确率 > 90%）。
4. 包含代词的模糊查询经重写后能正确召回目标记忆。
5. 后台补漏提取能捕获 Agent 遗漏的有效个人信息，且不产生重复记忆。
6. 行为模式检测在自动提取关闭时仍能正常触发和积累。
7. 记忆置信度能根据用户的确认或矛盾信息动态调整。
8. 所有新增功能的单元测试覆盖率 > 80%。
