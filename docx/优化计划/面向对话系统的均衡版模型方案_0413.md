# 面向对话系统的均衡版模型方案（0413）

本文是在上一版 `Pre-Router + Capability` 技术方案基础上，专门针对**对话系统响应时间**进行的一次收敛。新的目标不是单纯继续压缩模型参数，而是在**效果、时延、可扩展性**三者之间找到更平衡的组合。因此，本版方案的核心原则是：**缩小常驻模型、保留可扩展召回、把重模块改成按需触发。**

| 模块 | 上一版方案 | 优化后建议 | 是否常驻 | 设计理由 |
|---|---|---|---|---|
| **Pre-Router** | Qwen3-1.7B-Instruct LoRA | **更小档位的 0.5B~0.6B 级 Instruct 模型**，配合规则优先 | **是** | 这是每轮最容易被触发的模块，应优先压缩 |
| **Capability 召回** | Qwen3-Embedding-0.6B | **保持 0.6B Embedding** | **是** | 要兼顾扩展性与召回质量，不宜过度缩小 |
| **Capability 精排** | Qwen3-Reranker-0.6B | **保留，但改为条件触发** | **否** | 只在候选冲突、低置信度或跨域时使用 |
| **Planner** | 通用 LLM | **保持现状，但仅复杂任务触发** | **否** | 不常驻，不进入普通对话轮次 |

## 一、优化后的总体判断

如果把系统目标定义为“像普通聊天一样顺滑，同时保留任务执行能力”，那么最合理的选择不是把整条链路全部缩到最小，而是把时延预算放在**触发频率最高的模块**上。按照这一原则，**Pre-Router 是第一优先级的缩减对象，Capability 不是。**

其原因在于，Pre-Router 本质上还是一个**固定标签空间下的粗路由问题**，它输出的是有限 schema，例如 `domain / memory_gate / planner_gate / required_agents`。这类任务更接近多头分类，适合用更小的 Instruct 模型甚至规则兜底处理。而 Capability 对应的是**能力语义检索**，它面对的是不断扩张的 Agent Card、MCP Server 与工具能力集合，因此更需要语义表征能力与开放召回能力，简单继续缩小会更容易直接伤到召回质量。[1] [2]

> **本版结论**：Pre-Router 可以再小一档；Capability 不建议继续简单缩模，而应通过“缓存 + 条件精排 + 候选池约束”来降时延。

## 二、推荐的均衡版分层方案

### 1. Pre-Router

Pre-Router 继续保留，但职责收缩为**轻量粗路由**。建议仅输出五个字段：`domain`、`complexity`、`memory_gate`、`planner_gate`、`required_agents`。其中最重要的是 `planner_gate` 与 `required_agents`，因为它们决定后续是否要进入重链路。

| 项目 | 优化建议 |
|---|---|
| **推荐模型** | **0.5B~0.6B 级中文/多语 Instruct 小模型** |
| **训练方式** | LoRA SFT，多头结构化输出 |
| **推理方式** | 低温度、严格 JSON schema |
| **运行策略** | **规则优先，模型回退** |

这里不再坚持 1.7B 的原因不是它不能用，而是它对对话系统来说更像“保守型方案”。如果每一轮都要先过路由器，那么 1.7B 会成为基础时延的一部分；而把它缩到 0.5B~0.6B 档，通常更符合“高频入口模块”的角色定位。

### 2. Capability 召回

Capability 仍建议采用**Embedding 召回**，不改成闭集分类器。原因没有变化：你们的 capability 来自 Agent Card，不是一个长期静态的标签表；未来增加 Agent、MCP 或工具时，继续沿用检索式方案的维护成本最低。[2] [3]

| 项目 | 优化建议 |
|---|---|
| **推荐模型** | **Qwen3-Embedding-0.6B** |
| **推理方式** | Top-K 召回，默认 K=5 或 K=8 |
| **缓存策略** | 对高频用户表达与 capability 结果做缓存 |
| **候选约束** | 先受 Pre-Router 的 `required_agents` 限制，再做向量召回 |

这一层虽然常驻，但它不应该面向全量 capability 池裸跑，而应该先用 Pre-Router 给出一个**候选边界**。例如用户被粗分到 `navigationAgent`，那么 Capability 只需在导航域的若干能力卡片里召回，而不是在所有域全局搜索。这样做对时延和准确率都会更友好。

### 3. Capability 精排

Reranker 不建议删除，但要从“默认步骤”改成“条件步骤”。也就是说，只有在以下情况出现时，才触发 Reranker：第一，Embedding 前两名得分接近；第二，Top-K 候选横跨多个域；第三，命中了高风险易混能力，例如“查文件”与“统计文件”“路线规划”与“地点搜索”。

| 触发条件 | 是否启用 Reranker |
|---|---|
| Top-1 与 Top-2 分差明显 | 否 |
| Top-1 置信度很高 | 否 |
| 候选冲突大 / 同域近义能力聚集 | 是 |
| 跨域混淆明显 | 是 |
| 用户表达模糊，需要 disambiguation | 是 |

这种设计的本质，是把 0.6B Reranker 从“全量串行开销”改成“边界样本修正器”。这样系统在绝大多数普通轮次里，只需完成 **Pre-Router + Embedding**，就能结束前置判断。

### 4. Planner

Planner 保持现有通用 LLM 方案，不建议在这一步缩模或训练。原因仍然是：Planner 不是高频入口，而是**复杂任务时才触发的长尾模块**。它的最优策略不是缩小，而是减少触发频率。

| 项目 | 优化建议 |
|---|---|
| **是否保留** | 保留 |
| **是否训练专门模型** | 暂不建议 |
| **触发策略** | 仅 `planner_gate=PLAN` 时触发 |
| **输入约束** | 接收已经过 Capability 收敛后的候选边界 |

## 三、对话系统中的推荐调用链

新的关键不只是模型大小，而是**默认调用链的长度**。对于普通对话系统，不能把所有重模块都挂在每一轮上。更合理的设计是把轮次分成三类。

| 轮次类型 | 示例 | 推荐链路 |
|---|---|---|
| **普通追问/闲聊轮次** | “继续”“展开讲讲”“这是什么意思” | 直接走对话回复，不重新全量路由 |
| **明确任务进入轮次** | “帮我找昨天下载的 PPT” | Pre-Router → Capability Embedding |
| **复杂或边界轮次** | “帮我整理本地文件并给出清理建议，再发我总结” | Pre-Router → Capability → 条件 Reranker → Planner |

> **这意味着，对话系统的时延优化重点应该是“减少重链路默认触发”，而不是“把每个模型都换成最小”。**

## 四、最终推荐配置

下面这张表可以直接作为你们后续实现的优先方案。

| 层级 | 推荐配置 | 角色定位 | 触发策略 |
|---|---|---|---|
| **Pre-Router** | **0.5B~0.6B Instruct 小模型 + 规则优先** | 高频粗路由 | 默认触发 |
| **Capability Embedding** | **Qwen3-Embedding-0.6B** | 常驻能力召回 | 默认触发，但受域边界约束 |
| **Capability Reranker** | **Qwen3-Reranker-0.6B** | 边界样本修正 | 条件触发 |
| **Planner** | 现有强 LLM | 复杂任务拆解 | 仅复杂任务触发 |

如果只保留一句工程判断，那么我建议写成：

> **面向对话系统时，采用“更小的 Pre-Router + 保持 0.6B Capability 检索 + 条件触发 Reranker + 非常驻 Planner”的均衡方案。**

这个版本相比上一版更适合真实在线对话，因为它把最昂贵的部分从“每轮必跑”改成了“只有必要时才跑”，同时保留了多 Agent 扩展能力。

## 五、落地优先级建议

| 优先级 | 动作 | 预期收益 |
|---|---|---|
| **P0** | 将 Pre-Router 缩小一档，并保留规则优先 | 立刻降低基础轮次时延 |
| **P0** | Reranker 改为条件触发 | 显著减少串行时延 |
| **P1** | 为 Capability 召回增加缓存 | 降低高频重复请求成本 |
| **P1** | 用 `required_agents` 缩小召回池 | 同时提升速度与精度 |
| **P2** | 仅对复杂任务开放 Planner | 控制长尾延迟 |
